How I Built WakfuDB

Why I made it, what problems it solves, and how it's structured.

Published June 2026 · All blog posts

WakfuDB started as a personal frustration project. I was theorycrafting a build, had twelve wiki tabs open, and still couldn't answer a simple question: "Where does this resource drop, and what crafts use it?" Wikis are invaluable, but they're not optimized for mid-game lookup speed. WakfuDB is my attempt to fix that.

The Problem

Wakfu's systems are interconnected. Items link to recipes, recipes link to resources, resources link to zones and monsters. Wiki pages capture this, but navigation is page-heavy. Search is often slow or scattered across fan sites with different update schedules.

I wanted something closer to a traditional game database - type a name, get a clean result, see related entries without losing context.

Design Goals

Before writing code, I set a few constraints:

  • Fast search first - the homepage should get out of your way
  • Readable on mobile - I check databases on a phone next to my main screen
  • Cross-linked data - items should surface related monsters, recipes, and resources
  • No account wall - reference tools should be frictionless
  • Maintainable after patches - updates shouldn't require rewriting the whole app

Data Pipeline

Game data changes with every patch. WakfuDB needs a pipeline that can ingest updated definitions, normalize them into a consistent schema, and publish without long downtime.

The core approach is separating data from presentation. Raw game definitions get transformed into JSON or database records with stable IDs. The frontend only cares about rendering entities and links - not where the XML or extracted files originally lived.

When Ankama ships a patch, I update the source data, run the import scripts, verify a handful of known items and monsters, and deploy. Community reports help catch edge cases - renamed items, deprecated recipes, new resource nodes.

Frontend Experience

The UI is intentionally boring in a good way: search bar, filters, detail pages. Players don't need flashy animations when they're trying to confirm whether a helmet requires level 110 or 115.

Detail pages group information by what players actually ask:

  • Stats and requirements for gear
  • Drop sources and locations for resources
  • Related recipes and crafting chains
  • Monster families and linked zones

Each section cross-links to other entities in the database so you can click through a crafting chain without returning to search.

Performance Considerations

Encyclopedia sites can get slow when search scans huge JSON blobs on every keystroke. WakfuDB uses indexed search - either pre-built indices or a lightweight server query - so results feel instant even with thousands of entries.

Images and icons are cached aggressively. Most players revisit the same small set of items during a session; cache headers and CDN delivery keep repeat loads snappy.

What I'd Do Differently

If I started over, I'd invest even earlier in automated patch diffing - scripts that compare old and new data exports and flag changed IDs automatically. Manual verification doesn't scale forever, especially when balance patches touch hundreds of items.

I'd also document the data schema publicly so contributors know how to propose fixes without asking me first.

What's Next

Ongoing work includes better filter options, build planner experiments, and faster patch turnaround. If you use WakfuDB regularly, the best way to help is reporting incorrect entries with specifics - item name, what you expected, and what the game actually shows.
